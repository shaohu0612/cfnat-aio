# CFNAT-AIO 产品版本迭代方案

## 1 设计原则

**兼容优先**：所有功能迭代必须基于现有 WebUI 的 7 个 tab（仪表盘、地区管理、cfnat 配置、cfdata、IP 库、日志、通用设置）进行扩展，不新增 tab 页、不删除现有功能、不改变现有 tab 的核心布局。新功能以"增加列、增加卡片、增加折叠区域"的方式融入现有页面，用户升级后原有操作习惯不受影响。

**WebUI 可配**：所有代理转发相关参数——负载均衡策略、重试次数、淘汰阈值、质量监控参数——必须在"通用设置"tab 中可配置、可查看，用户不需要修改代码或配置文件。所有参数变更实时生效，无需重启容器。

**渐进式展示**：新版本增加的列和指标默认折叠或隐藏，用户可通过列设置开关手动开启。已有的列和操作保持不变，确保现有功能零回归。

**哪里发生哪里看到**：负载分布放在仪表盘（地区卡片级），IP 质量放在 IP 库（行级），全局参数放在通用设置（配置区域级）。

## 2 现状诊断

### 2.1 IP 选取：纯随机，无智能调度

`pickTarget()` 每次从 IP 池中 `rand.Intn()` 随机选取一个 IP，不区分冷热、不参考速度/延迟。虽然 IP 库内部按速度降序排列，但选取时完全忽略这个排序。这意味着一个 40 MB/s 的 IP 和 0.5 MB/s 的 IP 被选中的概率相同。

### 2.2 连接失败：无重试，直接断开

`handleConn()` 连接上游失败后仅记录 `fail_count + 1`，随后立即 `return` 断开客户端连接。客户端体验为"代理不可用"，而实际上 IP 池中可能有数百个可用 IP。

### 2.3 淘汰机制：被动触发，无自动调度

`fail_count` 只在两处递增：`HealthCheck` 探活失败、`handleConn` 上游 TCP 连接失败。`RemoveFailingIPs(3)` 阈值硬编码为 3。系统没有内置定时健康检查调度，依赖外部调用。这意味着一个 IP 在转发过程中持续工作但质量下降（延迟升高、丢包增加），系统无法感知。

### 2.4 架构盲区

| 能力 | 现状 | 影响 |
|------|------|------|
| 连接级负载均衡 | 无 | 热点 IP 过载，其他 IP 闲置 |
| 转发质量监控 | 无 | 延迟升高、丢包无法感知 |
| 连接亲和性 | 无 | 短连接频繁切换 IP 带来 TLS 握手开销 |
| 故障转移 | 无 | 一次失败即断开客户端 |
| 实时降级 | 无 | 质量劣化 IP 继续参与转发 |

## 3 行业参考思路

同类 Cloudflare 代理工具在负载均衡方面的设计思路值得借鉴，核心要点如下：

**双池分离架构**：将 IP 池拆为主池（活跃转发）和备选池（待命补充）。主池容量有限，只放经过验证的最优 IP；备选池从 IP 库中按速度排序持续补充。主池节点被淘汰时，从备选池头部提升最优节点，保证替换无缝衔接。

**最少连接数选择**：每次选取时随机取两个候选 IP，比较当前活跃连接数，选择更少者。这种"两次随机取最优"的变体在理论和实践上都能以极低开销实现近似最优的负载均衡，同时通过原子计数保持无锁高性能。

**指数加权移动平均（EWMA）**：对每个 IP 的实时延迟和丢包率使用 EWMA 平滑，避免单次异常值导致误判。衰减因子 α = 0.04（约 50 个样本的半衰期），兼顾灵敏度和稳定性。评分时将延迟和丢包率融合为单一性能指标：`perf = delay * (1 + loss * 2)`，丢包率权重为延迟的 2 倍。

**连接亲和性（Sticky Sessions）**：短连接场景（如 HTTP 浏览）中，同一客户端在短时间内复用同一后端 IP，可省去重复的 TLS 握手。通过客户端 IP 哈希绑定后端，设置 15 秒 TTL，到期后自动轮换到连接数最少的新后端。

**转发层重试与故障转移**：连接上游失败后不直接断开客户端，而是换一个 IP 重试（最多 3 次）。每次重试释放前一个 IP 的连接计数，记录失败，并触发驱逐评估。3 次全失败才返回错误。

**定时健康检查 + 隔离恢复**：后台定期对 IP 池执行 TCP 探活。质量劣化的 IP 先标记为"隔离"状态（暂时不参与转发），保留恢复机会。隔离超过一定时间仍未恢复则永久移除。新入库 IP 有预热保护期，避免样本不足时被误判淘汰。

## 5 版本迭代路线图

### V1.1 — 基础重试与故障转移（优先级：P0）

**目标**：解决"一次失败即断开"的核心体验问题。

**后端改动**：`internal/proxy/manager.go` — `handleConn()` 函数

**具体方案**：

1. 在 `handleConn()` 中增加重试循环，最多 N 次（N 从配置读取）
2. 每次重试前调用 `pickTarget()` 重新选取 IP（引入 `retryExclude` 集合排除已失败的 IP）
3. N 次全部失败后返回错误给客户端，同时记录该地区所有尝试 IP 的失败计数
4. 重试之间无需额外延迟（上游 TCP 连接失败本身就消耗了等待时间）

**WebUI 落地**：

| tab | 变更 | 类型 |
|-----|------|------|
| settings | "代理转发"折叠区新增"最大重试次数"滑块（0-5，默认 3） | 新增配置项 |
| dash | 地区卡片内新增"今日重试"数字（如 `Retry: 12`），紧跟在现有"可用 IP"行之后 | 新增指标行 |
| logs | 重试事件以 `[proxy]` 来源标签输出，复用现有日志筛选 | 新增日志来源 |

**兼容性保障**：
- 仪表盘地区卡片现有字段不受影响，新行追加在末尾
- 如果重试次数为 0（用户主动关闭），行为与当前版本完全一致
- settings 中"代理转发"折叠区默认收起，不干扰现有设置

---

### V1.2 — 最少连接数负载均衡（优先级：P0）

**目标**：避免热点 IP 过载，均衡利用所有可用 IP。

**后端改动**：`internal/proxy/manager.go` — 新增 `connCounts sync.Map`，`pickTarget()` 函数，`Status()` API 扩展

**具体方案**：

1. 在 `Manager` 结构体中新增 `connCounts sync.Map`（key=IP string, value=`*atomic.Int32`），实现无锁连接计数
2. `pickTarget()` 改为从可用 IP 中随机取两个位置（index 和 index + len/2），比较 `connCounts`，返回连接数更少者
3. 连接建立成功后 `connCounts[ip].Add(1)`，连接关闭时（defer）`connCounts[ip].Add(-1)`
4. IP 被删除时从 `connCounts` 中 `Delete` 对应 key
5. 如果两个候选 IP 连接数相同，选择速度更高的（利用 IP 库已有的速度降序缓存）

**WebUI 落地**：

| tab | 变更 | 类型 |
|-----|------|------|
| settings | "代理转发"折叠区新增"负载均衡策略"下拉框（纯随机/最少连接数/加权随机（灰显，V1.5 解锁）），默认最少连接数 | 新增配置项 |
| dash | 地区卡片内新增"活跃连接"数字，紧跟在"可用 IP"行之后 | 新增指标行 |
| ips | 表格列设置中新增"连接数"开关（默认关闭），开启后显示该 IP 当前活跃连接数，数字旁带小型进度条 | 新增可隐藏列 |

**兼容性保障**：
- 默认策略为"最少连接数"，后端行为改变但对外接口不变
- 用户可随时切回"纯随机"恢复当前版本行为
- IP 库"连接数"列默认隐藏，不影响现有表格布局

### V1.3 — 转发质量实时监控（优先级：P1）

**目标**：感知每个 IP 在转发过程中的实际表现，为淘汰决策提供数据基础。

**后端改动**：`internal/proxy/manager.go` — 新增 `IPMetrics` 结构体和采集逻辑，`internal/iplibrary/library.go` — `Status()` API 扩展

**前置依赖**：无

**具体方案**：

1. 新增 `IPMetrics` 结构体：
   ```go
   type IPMetrics struct {
       ewmaDelay    float64  // 指数加权平均延迟（ms）
       ewmaLoss     float64  // 指数加权平均丢包率（0-1）
       sampleCount  int      // 样本计数
       lastActive   time.Time
       mu           sync.RWMutex
   }
   ```
2. 在 `Manager` 中维护 `metrics map[string]*IPMetrics`，key 为 IP
3. 在双向转发（`io.Copy`）中采集指标：
   - 转发完成且无错误：记录延迟样本，`ewmaLoss` 不变
   - 转发出错：`ewmaLoss` 加权更新
   - 延迟测量：记录客户端连接建立到第一次数据返回的时间差
4. EWMA 更新公式：`new = old * 0.96 + sample * 0.04`
5. `Status()` API 输出中每个地区增加 `avg_delay_ms`、`avg_loss_rate` 字段

**WebUI 落地**：

| tab | 变更 | 类型 |
|-----|------|------|
| settings | "代理转发"折叠区新增"EWMA 样本窗口"滑块（20-200，默认 50） | 新增配置项 |
| dash | 地区卡片内新增"延迟"和"丢包"两个数值，延迟用颜色标签（绿<200ms、黄200-500ms、红>500ms），丢包用百分比显示 | 新增指标行 |
| ips | 表格列设置中新增"质量评分"开关（默认关闭），开启后显示 0-100 综合评分，带颜色标签 | 新增可隐藏列 |
| ips | 表格列设置中新增"EWMA 延迟"和"EWMA 丢包率"开关（默认关闭） | 新增可隐藏列 |

**兼容性保障**：
- 仪表盘新增指标行追加在卡片末尾，不改变现有字段
- IP 库所有新增列默认隐藏，用户手动开启后才显示
- 未采集到样本的 IP 显示"--"而非空白，避免视觉空洞

### V1.4 — 主动淘汰与自动健康检查（优先级：P1）

**目标**：变被动淘汰为主动淘汰，系统自动感知并移除劣化 IP。

**后端改动**：`internal/proxy/manager.go` — 新增 `healthCheckLoop` goroutine，`internal/iplibrary/library.go` — 新增 `RemoveByQuality` 方法

**前置依赖**：V1.3（需要 EWMA 监控数据）

**具体方案**：

1. 在 `Manager.Start()` 中启动 `healthCheckLoop` goroutine，间隔从配置读取
2. 检查逻辑：
   - 遍历所有 IP 的 `IPMetrics`
   - `ewmaDelay > MaxDelayMs` 且 `sampleCount >= 20` → 标记待淘汰
   - `ewmaLoss > MaxLossRate` 且 `sampleCount >= 20` → 标记待淘汰
   - 待淘汰 IP 先标记为"隔离"状态（不参与 `pickTarget`），保留隔离时长
   - 隔离期满仍未恢复 → 从 IP 库永久移除
3. 转发层即时淘汰：`handleConn` 中上游连接失败时，`failCount >= 3` 立即移除
4. 新入库 IP 在预热保护期内免于被隔离

**WebUI 落地**：

| tab | 变更 | 类型 |
|-----|------|------|
| settings | "代理转发"折叠区新增 5 个配置项：健康检查间隔（30-600s，默认120）、最大延迟阈值（100-2000ms，默认500）、最大丢包率阈值（1-50%，默认10）、隔离时长（60-900s，默认300）、预热保护期（0-300s，默认60） | 新增配置项 |
| ips | 表格列设置中新增"状态"开关（默认开启），显示 IP 当前状态：正常（绿色）、隔离中（橙色+倒计时）、预热中（蓝色） | 新增列（默认开启） |
| ips | 隔离状态的 IP 行背景变为浅橙色，点击操作列的"详情"可查看隔离原因和剩余时间 | 新增行样式 |
| dash | 地区卡片顶部新增健康检查状态指示灯（绿=正常、黄=检查中、红=发现劣化 IP） | 新增指示灯 |
| logs | 健康检查结果以 `[health]` 来源标签输出，如 `HealthCheck: 3 IPs isolated, 1 IP removed` | 新增日志来源 |

**兼容性保障**：
- 健康检查默认启用（间隔 120s），如果 EWMA 样本不足（V1.3 未部署），则仅依赖 fail_count 淘汰，行为等价于当前版本
- IP 库"状态"列默认开启（仅显示正常/预热中，隔离中按需出现），不改变现有列顺序
- 隔离 IP 不从表格中消失，用户仍可查看和手动操作

### V1.5 — 加权随机选取（优先级：P2）

**目标**：在"最少连接数"基础上，进一步偏向速度更快、延迟更低的 IP。

**后端改动**：`internal/proxy/manager.go` — `pickTarget()` 函数改造

**前置依赖**：V1.3（加权公式中的延迟数据）

**具体方案**：

1. 对候选 IP 池中的每个 IP 计算权重：
   ```
   weight = speed_mbps / (1.0 + ewma_delay_ms / 100.0)
   ```
2. 速度数据来自 IP 库中的 `speed_mbps` 字段，延迟来自 V1.3 的 `IPMetrics.ewmaDelay`
3. 无测速数据的 IP 赋予默认权重 1.0
4. `pickTarget()` 改为两步选取：
   - 第一步：随机取 3 个候选 IP
   - 第二步：按连接数过滤（如果有两个连接数明显更少，优先选少的）
   - 第三步：在连接数接近的候选中，按权重随机选 1 个
5. 权重随机实现：累加所有权重生成 [0, total) 区间，随机取点按区间命中

**WebUI 落地**：

| tab | 变更 | 类型 |
|-----|------|------|
| settings | "负载均衡策略"下拉框中"加权随机"选项解除灰显，选中后下方显示公式说明文字 | 修改配置项 |
| ips | 表格列设置中新增"权重"开关（默认关闭），仅在加权随机策略时有效，显示该 IP 在池中的权重占比 | 新增可隐藏列 |

**兼容性保障**：
- 不改变最少连接数策略的任何行为
- "加权随机"只是策略选项之一，用户可随时切回其他策略
- "权重"列仅在对应策略时显示，其他策略下即使开启也不显示

### V1.6 — 连接亲和性（优先级：P2）

**目标**：减少短连接频繁切换 IP 带来的 TLS 握手开销。

**后端改动**：`internal/proxy/manager.go` — 新增 StickySlot 机制，`Status()` API 扩展

**前置依赖**：无（但建议在 V1.2 之后，因为依赖连接数判断）

**具体方案**：

1. 新增 `StickySlot` 结构体：
   ```go
   type StickySlot struct {
       clientHash  string
       backendIP   string
       lastAccess  time.Time
       createdAt   time.Time
   }
   ```
2. 在 `Manager` 中维护 `stickyMap map[string]*StickySlot`，key 为 `clientIP:region`
3. `handleConn()` 中先检查是否有有效 StickySlot（TTL 从配置读取）
4. 有有效槽位 → 验证该后端仍在 IP 库且连接数未过高 → 直接使用
5. 无有效槽位 → 走正常选取流程 → 创建新槽位
6. 后台每 10 秒清理过期槽位
7. 最大槽位数限制为 IP 池大小的 50%

**WebUI 落地**：

| tab | 变更 | 类型 |
|-----|------|------|
| settings | "代理转发"折叠区新增"连接亲和性"开关（默认关闭）和"亲和性 TTL"滑块（5-60s，默认 15，开关关闭时禁用） | 新增配置项 |
| dash | 地区卡片内新增"Sticky"数字（当前活跃槽位数），默认不显示（亲和性关闭时），开启后显示 | 条件新增指标行 |
| ips | IP 库筛选区域新增"绑定客户端"筛选选项（仅在亲和性开启时显示），可查看某 IP 当前绑定的客户端数量 | 条件新增筛选 |

**兼容性保障**：
- 连接亲和性默认关闭，后端不创建任何 StickySlot，行为等价于当前版本
- 所有 Sticky 相关 UI 均为条件显示，亲和性关闭时用户完全感知不到这些功能的存在

### V1.7 — 双层 IP 池（优先级：P3）

**目标**：将 IP 库拆分为主池和备选池，提升转发稳定性和替换速度。

**后端改动**：`internal/iplibrary/library.go` — 重构 `Library` 为双层结构，`internal/proxy/manager.go` — 池间提升逻辑

**前置依赖**：V1.2（最少连接数）、V1.4（主动淘汰）

**具体方案**：

1. `Library` 新增两个池：
   ```go
   activePool   []IPEntry  // 主池，容量 N（可配置，默认 20）
   standbyPool  []IPEntry  // 备选池，容量 ceil(N/2)
   ```
2. 主池用于 `pickTarget()` 转发，备选池定期健康检查并按综合评分排序
3. 主池节点被淘汰时，从备选池头部提升最优节点，同时触发扫描器补充备选池
4. 备选池容量不足时，从 IP 库全量中按速度降序筛选补充
5. 综合评分 = `speed_mbps * (1 - ewma_loss) / (1 + ewma_delay / 200)`
6. 新增配置项 `ActivePoolSize`（默认 20）、`StandbyPoolRatio`（默认 0.5）

**WebUI 落地**：

| tab | 变更 | 类型 |
|-----|------|------|
| settings | "代理转发"折叠区新增"主池容量"滑块（5-100，默认 20）和"备选池比例"滑块（0.2-1.0，默认 0.5） | 新增配置项 |
| ips | IP 库表格列设置中新增"所属池"开关（默认开启），显示"主池"或"备选池"标签 | 新增列 |
| ips | IP 库筛选区域新增"所属池"下拉筛选（主池/备选池/全部），默认"全部" | 新增筛选项 |
| dash | 地区卡片内"可用 IP"数字旁新增池容量标注，如 `441 (主池 20/20)` | 修改指标行 |
| logs | 池间提升事件以 `[proxy]` 来源标签输出：`[proxy] 主池淘汰 104.16.1.1, 提升 104.16.2.2 入主池` | 新增日志 |

**兼容性保障**：
- 主池容量默认 20，如果 IP 库中某地区 IP 少于 20 个，全部自动归入主池，行为等价于当前版本
- 备选池比例和容量调整即时生效，不影响已有连接
- IP 库表格的"所属池"列默认开启，但如果某地区 IP 少于主池容量，所有 IP 都显示"主池"，视觉上与当前版本无差异

|------|------|--------|----------|----------|

## 8 实施建议

**第 1 周：V1.1 + V1.2**（P0 快速修复）
- 后端改动并行开发
- WebUI 改动集中在 settings 折叠区和 dash 卡片新增行，不涉及结构变更
- 完成后部署运行 1-2 天观察稳定性

**第 2 周：V1.3 + V1.4**（P1 质量闭环）
- V1.3 是 V1.4 的前置条件，必须顺序开发
- WebUI 改动集中在 ips 表格新增列和 dash 卡片新增行
- 完成后部署运行 2-3 天，观察 EWMA 指标是否合理、淘汰是否准确

**第 3 周：V1.5 + V1.6**（P2 体验优化）
- 两者可独立开发
- WebUI 改动量最小（settings 解锁选项 + ips 条件显示列）
- 完成后部署运行 2-3 天

**第 4 周：V1.7**（P3 架构升级）
- 架构级重构，建议在前 6 个版本稳定运行后再启动
- 涉及 iplibrary 核心数据结构变更，需充分测试缓存重建和池间提升逻辑
- WebUI 改动集中在 ips 表格新增筛选和 dash 卡片数字标注

每个版本发布后运行观察期，确认无回归问题再进入下一版本开发。所有版本完成后，CFNAT-AIO 的代理转发能力将从"随机选 IP"进化为"智能负载均衡 + 自适应质量监控 + 自动淘汰"的完整闭环，且全部能力均通过现有 WebUI 的 7 个 tab 可配置、可监控，原有功能和布局不受影响。
