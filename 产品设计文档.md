# CFNAT-AIO 产品设计文档 v2.0

## 1 需求背景

CFNAT-AIO 是一个面向 Cloudflare CMIN2 线路优化的 All-In-One 工具，整合代理转发、IP 扫描与 WebUI 控制台于单一容器进程。

用户痛点：现有 cfnat 和 cfdata 是两个独立项目，分别负责代理转发和 IP 扫描。用户需要分别部署、分别配置、分别查看结果，维护成本高。CFNAT-AIO 将两者合并，通过统一的 WebUI 管理所有地区代理、IP 库和扫描任务，配置变更无需重启进程。

## 2 产品定位

| 维度 | 定义 |
|------|------|
| 目标用户 | 需要优化 Cloudflare 访问速度的技术用户（PT 下载、CDN 加速等场景） |
| 核心价值 | 多地区 SOCKS5/HTTP 代理 + CMIN2 IP 优选库，一键扫描、自动淘汰、WebUI 管理 |
| 部署形态 | 单容器单进程，Docker Compose 一键启动 |
| 典型用法 | qBittorrent/Transmission 通过 SOCKS5 连接不同地区端口，走对应 CMIN2 线路 |

## 3 功能需求详情

### 3.1 仪表盘

仪表盘是用户进入 WebUI 的首页，提供全局运行状态的一览视图。页面顶部显示系统运行时长和扫描任务状态。主体区域以卡片形式展示每个已启用地区的实时数据：地区名称、监听端口、可用 IP 数量、当前活跃连接数、最近一次使用的转发 IP、以及监听状态指示灯。当某个地区 IP 数量为 0 时，卡片会以视觉提示提醒用户该代理将 fallback 到全量 CF IP 池。

**新增特性**：
- 健康状态指示灯（绿色=healthy、黄色=degraded、红色=down）
- 双层 IP 池显示（主池/备选池数量）
- 连接亲和性计数
- 统计卡片（4列：总 IP 数、活跃连接、隔离 IP、预热 IP）

页面底部是实时日志流，通过 Server-Sent Events 持续接收代理、扫描器、WebUI 三个来源的日志。日志支持按来源筛选（proxy/scanner/webui/system），并显示时间戳和级别标签。日志区域自动滚动，用户可暂停滚动查看历史记录。

| 字段 | 来源 | 说明 |
|------|------|------|
| 可用 IP 数 | `iplibrary.CountIPsByCodes(region.Code)` | 按地区配置的 colo 代码聚合统计 |
| 当前 IP | `proxy.currentIP[region.Name]` | 最近一次连接使用的上游目标 IP |
| 监听状态 | `proxy.listeners` 映射存在性 | 配置变更后由 `Sync()` 热重载 |
| 健康状态 | `proxy.Status()` 计算 | healthy/degraded/down |
| 主池/备选池数量 | `iplibrary` 双层池 | 活跃 IP 池和备选 IP 池大小 |

### 3.2 地区管理

地区管理模块负责配置多地区代理监听。每个地区对应一个独立的 TCP 监听端口，客户端连接该端口后，流量会被转发到该地区对应 CMIN2 IP 库中的随机 IP。地区配置包含以下字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 地区名称 | string | 是 | 显示名称，如"日本"、"美国" |
| Colo 代码 | string | 是 | 逗号分隔的 Cloudflare 数据中心简码，如 `KIX,NRT,OKA` |
| 监听端口 | int | 是 | 本地监听端口，如 1003 |
| 启用 | bool | 是 | 是否启动该地区的代理监听 |
| Fallback | bool | 否 | IP 库为空时是否自动 fallback 到全量 CF IP 池 |

**业务逻辑**：

- 新增地区时，系统校验端口是否冲突，通过后写入 SQLite 并调用 `proxy.Sync()` 启动监听
- 修改地区（端口或 colo 代码变化）时，`Sync()` 检测到配置差异，自动 stop 旧 listener、start 新 listener
- 删除地区时，对应 listener 被关闭，已有连接正常断开
- 启用/停用切换时，仅控制 listener 的启停，不删除配置

### 3.3 IP 库

IP 库是 CMIN2 优选 IP 的存储与查询中心。所有入库 IP 都经过 TCP 探活 + 测速验证，按 colo 代码分库存储。WebUI 的 IP 库页面以表格形式展示，支持按地区（地区名称映射）和来源（手动/自动扫描/批量导入）筛选。

**新增列**：

| 列 | 说明 |
|----|------|
| 状态 | normal/isolated/warming（健康检查状态） |
| 池 | active/standby/none（所属 IP 池） |
| 连接数 | 当前活跃连接数 |
| EWMA 延迟 | 指数加权平均延迟 |
| EWMA 丢包率 | 指数加权平均丢包率 |
| 权重 | 基于速度和延迟计算的权重值 |

**手动添加 IP**：用户输入 IP、Colo 代码、速度和延迟，系统直接入库，不经过验证。

**入库规则**：

- 扫描器入库：通过 TCP 探活 + TLS 握手 + /cdn-cgi/trace 获取 colo + 下载测速，速度 >= 阈值才入库
- 批量导入探测：粘贴 IP 列表（支持 `ip:port#注释` 格式），系统并发探活识别 colo，CMIN2 节点再测速，达标自动入库
- 手动添加：直接写入，不验证，适合用户从外部渠道获取的已知可用 IP

**缓存机制**：IP 库使用内存缓存 `cache map[string][]string`，按速度降序排列。每次增删改后调用 `reload()` 重建缓存。代理选取 IP 时从缓存读取，避免频繁查询数据库。

**双层 IP 池**：
- 主池（active_pool）：活跃转发 IP，按速度降序排列，容量可配置（默认 20）
- 备选池（standby_pool）：待命补充 IP，当主池 IP 被隔离时自动从备选池提升，容量为主池的一定比例（默认 0.5）

### 3.4 扫描任务

扫描任务模块继承 cfdata 的扫描管线，对 Cloudflare IP 段进行系统性的优选。扫描分为 5 个阶段：

| 阶段 | 任务 | 输出 |
|------|------|------|
| 1. 加载 CIDR | 从 Cloudflare 官网获取 IPv4/IPv6 CIDR 列表，失败时 fallback 到内置列表 | CIDR 字符串数组 |
| 2. 抽样 | 每个 CIDR 段按 `SamplesPer24` 参数抽取随机 IP | 候选 IP 列表 |
| 3. 探活 | TCP connect + TLS 握手 + 请求 `/cdn-cgi/trace` 提取 colo | ProbeResult（IP, colo, 延迟, 是否通过） |
| 4. 测速 | 对探活通过的 IP 按延迟排序取 top 100，下载 speed.cloudflare.com 测速文件（2MB） | 速度（MB/s） |
| 5. 入库 | 速度 >= `MinSpeedMBps` 阈值的 IP 按 colo 入库 | 入库统计 |

**扫描配置**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 启用后台扫描 | false | 是否按 Interval 自动循环扫描 |
| 扫描间隔 | 60 分钟 | 两次扫描之间的间隔 |
| 测速阈值 | 3.0 MB/s | 入库最低速度要求 |
| IP 类型 | 4 | 4=IPv4, 6=IPv6（默认 IPv4，Docker 环境不支持 IPv6） |
| 抽样数/24 | 100 | 每个 /24 抽取的 IP 数（1-255） |
| 最大延迟 | 500 ms | 探活阶段延迟超过此值的 IP 丢弃 |
| 并发数 | 100 | 探活并发线程数 |
| 仅 CMIN2 | true | 是否只保留 CMIN2 白名单节点的 IP |

**后台调度**：扫描器在独立 goroutine 中运行，支持手动触发"立即扫描"和"停止扫描"。扫描历史持久化到 SQLite，记录每次扫描的启动时间、完成时间、状态、候选数、通过数。

**IPv6 自动检测**：启动时自动检测 IPv6 连通性，如果用户配置了 IPv6 但环境不支持，自动切换到 IPv4 并记录警告日志。

### 3.5 批量导入探测

批量导入探测是快速将外部 IP 来源（如 FOFA、CF-Workers 脚本输出）纳入 IP 库的通道。用户将 IP 列表粘贴到文本框，系统执行以下流程：

1. **解析**：支持 `ip:port#注释` 和纯 IP 格式，提取 IP 和注释
2. **去重**：相同 IP 只保留一条
3. **并发探活**：使用与扫描器相同的 `ProbeBatch` 逻辑，并发探测每个 IP 的 colo
4. **CMIN2 识别**：只保留 colo 在 CMIN2 白名单中的 IP（HKG, SIN, NRT, KIX, LAX, SJC, SEA, FRA, AMS, LHR, TPE, ICN, MNL, BKK, MFM）
5. **测速**：对 CMIN2 候选 IP 并发测速，限制并发 30 避免 rate limit
6. **入库**：速度 >= 阈值的 IP 自动入库，返回详细的每条 IP 探测结果（colo、延迟、速度、是否入库）

用户可选择"自动入库"或仅探测查看结果后再手动决定。探测结果以表格形式返回，方便用户筛选。

### 3.6 代理转发

代理转发是系统的核心运行时组件。每个启用地区在对应端口监听 TCP 连接，支持自动协议检测：

| 首字节范围 | 检测协议 | 处理方式 |
|------------|----------|----------|
| `0x05` | SOCKS5 | 完整 SOCKS5 握手（无需认证）→ CONNECT 应答 → 双向转发 |
| `0x20-0x7E` | HTTP CONNECT | 解析 HTTP 请求 → 200 Connection Established → 双向转发 |
| 其他 | TLS/透传 | 直接转发到上游，不做协议解析 |

#### 3.6.1 IP 选取策略（V1.1~V1.5）

**V1.1 — 基础重试与故障转移**：连接上游失败后自动重试，最多 N 次（可配置，默认 3 次）

**V1.2 — 最少连接数负载均衡**：从 IP 池中随机取两个候选，选择连接数更少者

**V1.3 — EWMA 质量监控**：使用指数加权移动平均（EWMA）实时监控每个 IP 的延迟和丢包率

**V1.4 — 主动淘汰与自动健康检查**：后台定期健康检查，质量劣化 IP 先隔离后移除，新 IP 有预热保护期

**V1.5 — 加权随机选取**：根据 IP 速度和延迟计算权重，在候选池中加权随机选取
- 权重公式：`weight = speed_mbps / (1.0 + ewma_delay_ms / 100.0)`

#### 3.6.2 连接亲和性（V1.6）

同一客户端短时间内复用同一后端 IP，减少 TLS 握手开销。

- 亲和性槽位：`StickySlot` 结构，记录客户端标识和绑定的 IP
- TTL：可配置（默认 15s），过期后槽位自动清理
- 开关：可在设置中启用/禁用

#### 3.6.3 双层 IP 池（V1.7）

将 IP 库拆分为主池和备选池：

- 主池（active_pool）：活跃转发 IP，容量可配置（默认 20）
- 备选池（standby_pool）：待命补充 IP，容量为主池的一定比例（默认 0.5）
- 自动提升：主池 IP 被隔离时，自动从备选池提升最优 IP 到主池

**IP 选取流程**：
1. 首选从主池中按权重随机选取
2. 如果主池为空或所有 IP 被隔离，从备选池中选取
3. 如果 IP 库为空，且 `Fallback=true`，从全量 CF CIDR 池中每 /24 抽 1 个 IP（最多 50 个候选）随机选取

**淘汰机制**：
- `HealthCheck` 被调用时，对每个 IP 执行 TCP 探活
- 探活失败的 IP，`fail_count` 递增
- 超过延迟/丢包率阈值的 IP 被隔离到隔离池
- 隔离池中的 IP 在隔离时长到期后重新评估
- 新入库 IP 有预热保护期，在此期间不会被隔离

### 5.1 模块分层

### 5.2 关键设计决策

**配置热重载**：所有配置变更先写 SQLite，再更新 `config.Manager` 的内存对象。代理模块通过 `Sync()` 对比期望配置和当前 listener，自动增删改监听端口，无需进程重启。

**IP 库缓存**：`iplibrary.Library` 维护 `cache map[string][]string`，按速度降序。每次 `AddIP/RemoveIP/HealthCheck` 后调用 `reload()` 重建。代理选取时从缓存读取，避免 DB 查询。

**双层 IP 池**：`iplibrary.Library` 维护 `activePool` 和 `standbyPool`，按速度降序排列。`RebuildPools()` 根据配置的容量和比例重建双层池。

**EWMA 质量监控**：`proxy.IPMetrics` 使用指数加权移动平均实时计算延迟和丢包率，权重因子为 `2/(N+1)`，其中 N 为样本窗口大小。

**连接亲和性**：`proxy.StickyManager` 维护 `slots sync.Map`，键为客户端标识（源 IP:端口），值为绑定的 IP 和过期时间。独立 goroutine 定期清理过期槽位。

**日志系统**：内存环形 buffer（默认 1000 条），支持 SSE 实时推送和快照查询。同时输出到 stdout，保留 `docker logs` 能力。

**Go embed 部署**：WebUI 的 `index.html` 通过 `//go:embed templates/*` 编译进二进制，单文件部署。

### 5.3 代理转发流程

```
客户端连接 → 协议检测(SOCKS5/HTTP/TLS) → 选取IP(加权随机)
    ↓
TCP连接上游 → 双向转发 → 记录延迟/丢包 → 更新EWMA指标
    ↓
连接失败 → 重试(最多N次) → 仍失败则标记IP
    ↓
健康检查 → 超过阈值 → 隔离IP → 从备选池提升新IP
```

## 6 已知限制与后续方向

| 限制 | 现状 | 可能改进 |
|------|------|----------|
| IP 选取策略 | 加权随机 | 支持更多负载均衡算法（加权轮询、最小响应时间） |
| 连接失败重试 | 固定次数重试 | 指数退避重试策略 |
| 自动健康检查 | 定期检查 | 自适应检查间隔（根据当前负载动态调整） |
| 延迟/丢包监控 | EWMA 统计 | 增加 P90/P99 延迟统计 |
| 负载均衡 | 最少连接数/加权随机 | 支持基于 CPU/内存的动态权重 |
| 兜底策略 | 从 CIDR 随机抽 | 优先选历史可用 IP |
| 协议检测 | 首字节检测 | 增加更完善的 SOCKS5 错误处理 |
| 监控告警 | 无 | 增加 Prometheus 指标输出 |
